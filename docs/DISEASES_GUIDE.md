# Правила додавання нових хвороб

Цей документ описує стандарт додавання нових хвороб у вкладку "Хвороби".

Мета модуля — допомогти лікарю швидко сформувати діагноз, довідкову підказку за МКХ-10 та короткі редаговані рекомендації. Модуль не замінює клінічне рішення лікаря.

## 1. Основний принцип

Кожна нова хвороба додається як окрема картка / модуль.

Якщо хвороба вже має окрему картку, її не потрібно дублювати як "супутній стан" в інших конструкторах.

Наприклад:

* гіпертонічна хвороба;
* ІХС;
* серцева недостатність;
* фібриляція передсердь.

Ці діагнози додаються окремими фрагментами у загальне поле діагнозу. Лікар сам вирішує, який діагноз є основним, а який супутнім.

## 2. Структура файлу хвороби

Кожна хвороба має зберігатися як окремий структурований об'єкт.

Базова структура:

```js
export const disease = {
  id: '',
  category: '',
  title: '',
  description: '',
  diagnosticOrientations: [],
  builder: {},
  icd10Hints: [],
  recommendationGroups: {},
};
```

## 3. Обов'язкові поля

### id

Короткий технічний ідентифікатор англійською.

Приклад:

```js
id: 'hypertension'
```

### category

Клінічний розділ.

Приклад:

```js
category: 'Кардіологія'
```

### title

Назва хвороби, яку бачить користувач.

Приклад:

```js
title: 'Гіпертонічна хвороба'
```

### description

Коротке пояснення, для чого ця картка.

Приклад:

```js
description: 'Практичний шаблон для швидкого формулювання діагнозу, плану дообстеження та рекомендацій.'
```

## 4. Діагностичні орієнтири

Блок має бути коротким.

Писати тільки те, що допомагає лікарю швидко зорієнтуватися:

* як запідозрити хворобу;
* що підтверджує діагноз;
* що важливо уточнити перед формулюванням.

Не потрібно писати повний підручниковий опис.

Приклад:

```js
diagnosticOrientations: [
  'Повторно підвищений офісний АТ >=140/90 мм рт. ст. або підтвердження за домашнім моніторингом / ДМАТ.',
  'Оцінити ступінь підвищення АТ, стадію, серцево-судинний ризик, ураження органів-мішеней і супутні захворювання.',
]
```

## 5. Конструктор діагнозу

Конструктор має допомагати зібрати фразу, але не забирати контроль у лікаря.

Можливі поля:

* форма / тип хвороби;
* стадія;
* ступінь;
* функціональний клас;
* фенотип;
* ризик;
* клінічні прояви;
* ускладнення;
* ручне поле "додатково".

Обов'язково залишати можливість ручного доповнення.

## 6. Загальне поле діагнозу

Кнопка "Додати до діагнозу" додає сформований фрагмент у загальне поле.

Правила:

* попередній діагноз не стирається;
* новий фрагмент додається нижче або після попереднього тексту;
* лікар вручну редагує порядок, первинність і фінальне формулювання;
* програма не вирішує, який діагноз основний.

## 7. МКХ-10

МКХ-10 показується тільки як довідкова підказка.

Це не автоматичне остаточне кодування.

Формат:

```js
icd10Hints: [
  {
    code: 'I10',
    label: 'Есенціальна гіпертензія',
    when: 'Неускладнена артеріальна гіпертензія без уточнених ускладнень.',
  },
]
```

Якщо є кілька можливих кодів, потрібно показувати їх як варіанти з коротким поясненням.

## 8. Рекомендації

Для кожної хвороби додаються короткі рекомендації у структурі:

```js
recommendationGroups: {
  labs: [],
  instrumental: [],
  consultations: [],
  lifestyle: [],
  medications: [],
}
```

Рекомендації мають бути короткими і редагованими.

Це нагадування для лікаря, а не автоматичне лікування.

## 9. Захист від дублів

Кожен пункт рекомендацій повинен мати стабільний `id`.

Приклад:

```js
{
  id: 'ecg',
  text: 'ЕКГ',
}
```

Якщо ЕКГ є в кількох хворобах, у сформованих рекомендаціях вона має з'явитися один раз.

Однакові пункти в різних хворобах повинні мати однаковий `id`.

Приклади спільних `id`:

```js
{ id: 'cbc', text: 'ЗАК' }
{ id: 'urinalysis', text: 'ЗАС' }
{ id: 'creatinine-egfr', text: 'креатинін/eGFR' }
{ id: 'electrolytes', text: 'електроліти (натрій, калій)' }
{ id: 'lipid-profile', text: 'ліпідограма' }
{ id: 'ecg', text: 'ЕКГ' }
{ id: 'echo', text: 'ЕхоКГ' }
{ id: 'smoking-cessation', text: 'відмова від куріння' }
{ id: 'physical-activity', text: 'дозовані фізичні навантаження' }
```

## 10. Стиль медикаментозних рекомендацій

Писати групи препаратів, а не торгові назви.

Бажаний формат:

```js
{
  id: 'beta-blockers-angina',
  text: 'бета-блокатори (контроль ЧСС, зменшення симптомів стенокардії)',
}
```

Не потрібно писати персоналізовані призначення, конкретні дози або схеми без прямого запиту.

## 11. Заборонено

При додаванні нової хвороби не можна:

* додавати OpenAI;
* додавати авторизацію;
* додавати базу даних;
* автоматично призначати лікування;
* дублювати хвороби, які вже мають окрему картку;
* використовувати торгові назви в рекомендаціях;
* писати великі підручникові тексти;
* подавати МКХ-10 як остаточне автоматичне кодування.

## 12. Перевірка після додавання нової хвороби

Після додавання нової хвороби потрібно перевірити:

1. Хвороба з'являється у вкладці "Хвороби".
2. Діагностичні орієнтири відкриваються коректно.
3. Конструктор формує зрозумілий фрагмент діагнозу.
4. Кнопка "Додати до діагнозу" не стирає попередній текст.
5. МКХ-10 додається як довідкова підказка.
6. Кнопка "Сформувати рекомендації" створює короткий шаблон.
7. Однакові рекомендації не дублюються.
8. Проходить `npm run build`.

## 13. Шаблон для нової хвороби

Перед реалізацією нової хвороби потрібно спочатку показати користувачу коротку картку:

```md
## Назва хвороби

### Діагностичні орієнтири
- ...
- ...
- ...

### Поля конструктора
- ...
- ...
- ...

### МКХ-10 підказки
- ...
- ...

### Рекомендації

Дообстеження:
- лабораторні: ...
- інструментальні: ...
- консультації: ...

Спосіб життя:
- ...

Медикаментозна терапія:
- ...

### Що не дублюємо
- ...
```

Після узгодження з користувачем можна вносити зміни в код.
