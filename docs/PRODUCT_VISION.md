# Product Vision

## Назва продукту

Асистент лікаря

## Загальне бачення

Асистент лікаря має стати сучасною українською платформою для щоденної роботи лікаря: з калькуляторами, опитувальниками, структурованими формами, клінічними підказками та майбутньою безпечною AI-допомогою.

Продукт не повинен замінювати лікаря. Його роль — зменшити рутину, пришвидшити прийом, покращити структуру клінічних даних і допомогти лікарю працювати точніше.

## Бачення на 3-5 років

Через 3-5 років Асистент лікаря має перетворитися з набору окремих інструментів на повноцінний медичний SaaS для лікарів.

Очікуваний стан продукту:

* окремі SEO-сторінки для ключових калькуляторів і шкал;
* зручний робочий кабінет лікаря;
* безпечне локальне або контрольоване збереження даних;
* AI-допомога для підготовки чернеток заключень;
* модулі для різних спеціальностей;
* якісна українська медична база знань;
* стабільна аудиторія лікарів;
* прозора модель монетизації.

## Короткострокові цілі

Короткостроковий горизонт: найближчі тижні та місяці.

Цілі:

* стабілізувати поточний інтерфейс;
* покращити мобільну зручність;
* створити окремі SEO-сторінки для основних калькуляторів;
* додати короткі медичні пояснення до інструментів;
* перевірити індексацію в Google Search Console;
* зібрати перший фідбек від лікарів;
* не ускладнювати продукт авторизацією, базою даних або оплатою до реальної потреби.

## Середньострокові цілі

Середньостроковий горизонт: 6-18 місяців.

Цілі:

* розширити набір клінічних калькуляторів;
* додати більше опитувальників;
* створити сторінки з поясненнями шкал і клінічного використання;
* покращити структуру форми "Асистент лікаря";
* додати експорт або друк результатів, якщо це потрібно користувачам;
* протестувати локальне збереження чернеток без серверної бази;
* підготувати архітектуру для майбутнього AI-сервісного шару;
* сформувати першу активну аудиторію лікарів.

## Довгострокові цілі

Довгостроковий горизонт: 18 місяців і більше.

Цілі:

* створити кабінет лікаря, якщо буде підтверджена потреба;
* додати безпечне збереження даних з урахуванням медичної приватності;
* реалізувати AI-генерацію чернеток заключень через окремий сервісний шар;
* додати RAG на базі клінічних настанов після окремого рішення;
* розширити продукт на інші спеціальності;
* створити платні професійні функції;
* зберегти простоту MVP навіть після розширення функцій.

## Принципи розвитку

Продукт розвивається за такими принципами:

* спочатку користь для лікаря, потім технології;
* простота важливіша за кількість функцій;
* медичні дані не повинні передаватися на сервер без чіткої потреби;
* кожна нова функція має скорочувати рутину або покращувати якість роботи;
* калькулятори та формули не змінюються без прямого медичного рішення;
* AI не приймає клінічних рішень замість лікаря;
* інтерфейс має залишатися швидким і зрозумілим;
* SEO-розвиток не повинен шкодити якості медичного змісту.

## Критерії додавання нових функцій

Нову функцію варто додавати, якщо вона:

* потрібна лікарю в реальному робочому сценарії;
* зменшує кількість ручної роботи;
* має зрозуміле клінічне призначення;
* не створює зайвих ризиків для медичних даних;
* не перевантажує інтерфейс;
* може бути перевірена користувачами;
* не потребує складної інфраструктури без доведеної користі.

Функцію варто відкласти, якщо вона:

* додається лише тому, що це технологічно цікаво;
* потребує авторизації, бази даних або платежів без нагальної потреби;
* може заплутати лікаря;
* створює ризик неправильного медичного трактування;
* суперечить принципу простого робочого інструменту.

## AI стратегія

AI-функції відкладені до стабілізації основного продукту.

Майбутня AI-стратегія:

* AI має працювати через окремий сервісний шар;
* AI має створювати чернетки, а не фінальні медичні рішення;
* лікар завжди редагує і підтверджує результат;
* медичні дані не передаються зовнішнім сервісам без окремого рішення;
* перед підключенням AI треба визначити правила безпеки, приватності та зберігання даних;
* RAG на базі клінічних настанов додається тільки після окремого етапу проєктування.

Можливі AI-функції в майбутньому:

* генерація чернетки лікарського заключення;
* структурування введених даних;
* підказки щодо повноти заповнення форми;
* пояснення результатів калькуляторів для лікаря;
* пошук по клінічних настановах після реалізації RAG.

## Монетизація

Монетизація не є поточним етапом MVP.

Можливі майбутні моделі:

* безкоштовний базовий доступ до основних калькуляторів;
* платні професійні функції;
* підписка для лікаря;
* підписка для клініки;
* платний AI-модуль;
* експорт документів або шаблонів;
* командні функції для медичних закладів.

Принцип монетизації:

* базові медичні інструменти мають залишатися доступними;
* платними можуть бути функції, які економлять час, покращують організацію роботи або дають професійну автоматизацію;
* монетизація не повинна змушувати лікаря передавати медичні дані без потреби.

## Головний орієнтир

Асистент лікаря має зростати поступово: від простого корисного сайту до професійної медичної платформи, не втрачаючи швидкість, ясність і довіру лікаря.
